# NiNKU BOXX 人事制度プロダクト 要件定義書

## 1. プロダクト概要

### 1.1 プロダクトビジョン

ブルーカラー企業（解体業など）向けに、等級制度・評価制度の構築から運用までを一気通貫で支援するWebアプリケーション。現在スプレッドシートで手作業で行っている人事制度の設計・運用を、専用プロダクトに集約し、NiNKU BOXXのコンサルティングサービスと顧客企業双方の業務効率を大幅に改善する。

### 1.2 解決する課題

**顧客企業側の課題:**
- 昇給基準がブラックボックス化しており、従業員の不満・離職につながっている
- 育成目標やKPIが不在で、従業員の成長指標がない
- 上司による属人的評価が蔓延し、公平性に欠ける
- 360度評価の「誰が誰を評価するか」の設定・集計が煩雑

**NiNKU BOXX側（運用者側）の課題:**
- 各クライアント企業ごとにスプレッドシートで管理しており、制度変更時のインパクトが大きい
- 評価シートの印刷・配布・集計が手作業
- 等級と給与の紐付け確認が煩雑
- 従業員データと制度データの一元管理ができていない

### 1.3 ターゲットユーザー

| ユーザー種別 | 主な利用シーン |
|---|---|
| NiNKU BOXX運用者 | 制度設計、データ集計、ダッシュボード確認、レポート出力 |
| クライアント企業の経営者 | ダッシュボード閲覧、評価結果確認、昇格・昇給の意思決定 |
| クライアント企業の労務・人事担当 | 従業員データ管理、評価運用、給与シミュレーション |
| クライアント企業の上長（評価者） | 部下の評価入力、フィードバック記入 |
| クライアント企業の一般従業員 | 自己評価入力（360度評価回答） |

### 1.4 想定利用頻度

- 等級制度・号俸テーブルの設定：年1回（または制度改定時）
- 評価の実施：年2回（半期ごと）、ただし間の期間にも制度のアップデートや準備作業あり
- 従業員データの更新：随時
- ダッシュボード閲覧：随時

## 2. 機能要件

### 2.1 会社管理（マルチテナント）

プロダクトは複数のクライアント企業を管理するSaaS型を想定。

**会社情報の登録:**
- 会社名、所在地、代表者、設立日、事業内容
- 部署構成の設定
- 評価周期の設定（半期／年次）

**権限管理:**
- NiNKU BOXX管理者（全社横断アクセス）
- クライアント企業の管理者（自社のみアクセス）
- 評価者（評価入力権限）
- 一般従業員（自己評価・閲覧のみ）

---

### 2.2 従業員データベース

クライアント企業の全従業員の名簿・属性情報を一元管理する。

**基本属性:**
- 従業員ID
- 氏名（姓・名）
- 性別
- 生年月日
- 入社日
- 所属部署
- 雇用形態（正社員 / 契約社員 / パート）

**人事制度関連属性:**
- 職種大分類（例：現場 / 管理）
- 職種小分類（例：多能工 / ドライバー / オペレーター / 解体D）※等級との組み合わせで存在しないパターンがある場合は割当不可
- 等級（正1〜正6 など、会社ごとに定義）
- 役職（部長 / 次長・課長 / 職長 など）
- 現在の号俸
- 現在のランク
- 基本給（号俸テーブルから自動算出 or 手入力）

**ヒアリング・面談記録:**
- 初回面談時のモチベーション等のデータアップロード
- 面談日付と議事録のリンク（Google Driveなど外部保存を想定）
- 時系列での面談履歴管理

**パーソナルシート（個人詳細ページ）:**
- 上記の基本情報・人事情報をまとめて表示
- その人の職種・等級に基づく役割責任の表示
- 評価履歴の表示
- 給与情報の表示

### 2.3 等級制度管理

会社ごとに等級構造を定義・管理する。

#### 2.3.1 等級構造の設定

**等級レベルの定義:**
- 等級名（例：正1、正2、正3、正4、正5、正6）
- 雇用形態との対応（正社員 = 正1〜正5、契約社員 = 契約1〜契約4、パート = パ1〜パ3）
- 管理職 / 非管理職の区分
- 等級ごとの大分類（例：正1-正2 = 管理職、正3以下 = 非管理職）

**職種の定義:**
- 職種は階層構造（大分類 → 小分類）で定義可能
  - 例：「現場」大分類の下に「多能工」「ドライバー」「オペレーター」「解体D」などの小分類
  - 例：「管理」大分類の下に「経営管理職群」「技術専門職群」などの小分類
- 各職種に含まれる職位名
- 職種の追加・削除・名称変更・階層の変更はいつでも可能

**等級×職種の存在制御:**
- すべての等級×職種の組み合わせが存在するわけではない。企業や職種によって「その等級にはその職種が存在しない」ケースがある
- 例：「多能工」「ドライバー」は正1・正2が存在せず、正3以下のみ。「オペレーター」は正1が存在せず、正2以下のみ
- 各等級×職種の組み合わせごとに「有効/無効」を設定可能とする
- 無効（存在しない）に設定された組み合わせには、従業員を割り当てることができない
- 有効な組み合わせのみ、役割責任の入力・評価項目の設定が可能

| 等級＼職種 | 多能工 | ドライバー | オペレーター | 解体D |
|---|---|---|---|---|
| 正1（部門経営職） | ✕ 存在しない | ✕ 存在しない | ○ | ○ |
| 正2（業務管理職） | ✕ 存在しない | ✕ 存在しない | ○ | ○ |
| 正3（業務推進職） | ○ | ○ | ○ | ○ |
| 正4（指導職） | ○ | ○ | ○ | ○ |
| 正5（一般職） | ○ | ○ | ― | ― |
| 正6（補助職） | ○ | ― | ― | ― |

**⚠ 修正可能性に関する共通方針:**

等級・職種・号俸テーブル・ランク分割などの制度設定は、運用開始後もいつでも修正・追加・削除が可能な仕様とする。具体的には以下の通り。

| 設定項目 | 修正内容の例 | 対応方針 |
|---|---|---|
| 等級 | 等級数の増減、等級名の変更、管理職/非管理職区分の変更 | 既存従業員の等級との整合性チェックを行い、影響範囲を表示した上で反映 |
| 職種 | 職種の追加・削除・名称変更、階層構造の変更（大分類・小分類の追加等） | 該当職種に紐づく従業員がいる場合は警告表示、再割当を促す |
| 等級×職種の存在制御 | 特定の等級×職種の組み合わせを有効/無効に切り替え | 無効にする組み合わせに従業員が存在する場合は、先に従業員の再割当が必要（ブロック） |
| 号俸テーブル | 基本給MAX/MIN、増加率、初期号差などのパラメータ変更 | テーブルを再生成し、変更前後の差分プレビューを表示。既存の従業員当てはめへの影響を一覧で確認可能 |
| ランク分割 | ランク数の増減 | 号俸テーブル全体が再計算される。既存従業員のランク再マッピングが必要になる場合は警告表示 |
| 役割責任 | テキスト内容の変更、項目の追加・削除 | 即時反映。評価項目へのプリセット連動は次回評価期間から適用 |
| 役職 | 役職の追加・削除・名称変更 | 該当役職に紐づく従業員がいる場合は警告表示 |
| 評価基準マトリクス | S/A/B/C/Dの組み合わせロジック変更 | 過去の評価結果には影響せず、次回評価から適用 |
| 号俸改定基準 | 評価レート×ランクの改定値変更 | 過去の改定結果には影響せず、次回改定から適用 |

**修正時の共通ルール:**
1. **変更前後の差分プレビュー** — 設定変更を保存する前に、影響を受ける従業員数・給与変動額などの差分を表示する
2. **影響範囲の警告** — 従業員データに影響がある変更の場合、対象人数と具体的な影響内容を警告として表示する
3. **変更履歴の保持** — いつ・誰が・何を変更したかの履歴を記録する（監査対応）
4. **過去データの保全** — 制度変更は原則として「次回以降」に適用され、過去の評価結果や昇給実績を遡って変更しない
5. **段階的反映** — 号俸テーブルの再生成など大きな変更は、「プレビュー → 確認 → 確定」の3ステップで反映する

#### 2.3.2 役割責任の定義

等級×職種（小分類）ごとに、役割責任内容と該当者を管理する。

**各セル（等級×職種の組み合わせ）に含まれる情報:**
- 該当者一覧：その等級・その職種に属する従業員名（従業員データベースから自動反映）
- 役割責任内容：複数項目のテキスト入力（番号付きリスト形式）
- 主な職位名（例：業務推進職（上級職長・役職なし）、指導職（職長・役職なし）など）

**マトリクス表示:**
- 縦軸：等級（正1〜正6）× 職位名
- 横軸：職種（小分類単位。例：多能工、ドライバー、オペレーター、解体D）
- 各セルに「該当者」と「役割責任内容」を表示
- 存在しない組み合わせは「-」で表示し、入力不可

**表示例（現場系職種の場合）:**

| 等級 | 多能工 | ドライバー | オペレーター | 解体D |
|---|---|---|---|---|
| 正1（部門経営職） | - | - | 該当者+責任内容 | 該当者+責任内容 |
| 正2（業務管理職） | - | - | 該当者+責任内容 | 該当者+責任内容 |
| 正3（業務推進職） | 該当者+責任内容 | 該当者+責任内容 | 該当者+責任内容 | 該当者+責任内容 |
| 正4（指導職） | 該当者+責任内容 | 該当者+責任内容 | 該当者+責任内容 | 該当者+責任内容 |
| 正5（一般職） | 該当者+責任内容 | 該当者+責任内容 | - | - |
| 正6（補助職） | 該当者+責任内容 | - | - | - |

**役割責任内容の入力方式:**
- 各等級×職種の組み合わせごとに、5〜6項目程度の役割責任を入力
- 項目は番号付きで管理（1. 2. 3. ...）
- 項目の追加・削除・並び替えが可能
- 職種間で共通の責任内容がある場合も、職種ごとに個別に管理する（コピー機能は提供）
- PDF等での出力機能

### 2.4 号俸テーブル（給与テーブル）

等級ごとの基本給を号俸（ステップ）で管理するテーブル。

#### 2.4.1 テーブル生成パラメータ

以下のパラメータを入力すると、号俸テーブルが自動生成される。**すべてのパラメータは運用開始後もいつでも変更可能**であり、変更時はテーブルが再生成され、従業員への影響が差分プレビューとして表示される。

| パラメータ | 説明 | 例 |
|---|---|---|
| 基本給(MAX) | 全等級通じた基本給の上限額 | 600,000円 |
| 基本給(MIN) | 全等級通じた基本給の下限額 | 180,000円 |
| ランク分割 | 各等級内のランク数 | 8 |
| 増加率 | 号俸間の増加率 | 1.08 |
| 初期号差 | 最下位号俸の号差 | 2,000円 |
| ランク数 | 全体のランク数 | 15 |

#### 2.4.2 テーブル構造

- 縦軸：号俸番号（例：1〜120）+ ランク（例：1〜15、T付きは移行ランク）
- 横軸：等級（正1〜正6）
- セル値：基本給/月
- 各等級にはmax/minの基本給が設定され、ランク（S1〜D）に対応

**ランク体系（各等級共通）:**
S1, S2, S3, S4, S5, S6, S7, S8, A1, A2, A3, A4, A5, A6, A7, A8, B1, B2, B3, B4, B5, B6, B7, B8, C1, C2, ... , D1, D2, ...

#### 2.4.3 従業員の号俸当てはめ

- 従業員の等級と基本給から、号俸テーブル上の位置を自動判定
- 基本給と等級のミスマッチ検出（例：正3の等級なのに正4の給与レンジにいる場合にアラート）
- 手動での等級初期設定も可能
- 等級評価制度の結果に基づく等級設定も可能

#### 2.4.4 年収の算出

号俸テーブルの基本給/月 × 12ヶ月 = 年収として自動計算・表示。

### 2.5 評価基準・昇降格ロジック

#### 2.5.1 評価レート

評価結果をS/A/B/C/Dの5段階で表現する。前期と後期の評価の組み合わせにより、総合評価レートが決定される。

**評価基準マトリクス（前期＼後期）:**

| 前期＼後期 | S | A | B | C | D |
|---|---|---|---|---|---|
| S | S | S | B | C | C |
| A | A | S | A | B | C | C |
| B | B | B | B | C | C |
| C | C | B | C | C | C |
| D | D | C | C | C | D |

#### 2.5.2 等級別号俸改定基準

評価レートと現在のランクに基づいて、号俸の増減を決定する。

- 各等級（正1〜正6）ごとに、改定テーブルが存在
- 現賃金ランク（1〜15, T付き含む）× 評価レート → 号俸変動値（+N / -N）
- 等級が上位になるほど、同じランクで適用される改定幅の範囲が狭くなる

#### 2.5.3 昇格条件

以下の3条件を目安とする（あくまで目安であり、特別昇格もあり得る）：
1. 一定の号俸以上であること（各等級のAゾーン以上）
2. 総合人事評価が良いこと（直前年度でS評価、又は直前2年がA評価以上）
3. 専務・常務の承認が得られること

#### 2.5.4 降格条件

以下の3条件を目安とする：
1. 一定の号俸以下であること（各等級のBゾーン以下）
2. 総合人事評価が悪いこと（直前年度でD評価、又は直前2年がC評価以下）
3. 専務・常務の承認が得られること

### 2.6 評価制度

#### 2.6.1 個別評価（上司評価）

等級×職種ごとの役割責任をベースに、個別の評価項目を設定・運用する。

**評価シートの構成:**
- 対象期間（例：24年9月30日〜25年3月31日）
- 被評価者情報（氏名、評価者氏名、所属、等級）
- 評価カテゴリ（大項目）：計画・管理、安全への取り組み、その他
- 評価項目（中項目）：各カテゴリ内の具体的な評価項目
- 評価回答選択肢と説明：
  - 5点：非常に良い（常に達成）
  - 4点：良い（概ね達成）
  - 3点：普通（一部達成）
  - 1点：やや不十分（十分ではない）
  - 0点：不十分（達成できていない）
- 自己評価（今回）
- 前回評価（最終）
- 上司評価
- 自己評価フリーコメント（その点数をつけた理由）

**評価項目のプリセット機能:**
- 等級制度の役割責任で設定した項目が、評価項目のデフォルトとしてプリセットされる
- プリセットされた項目に対して、上書き・削除・追加が可能
- 評価周期ごとに項目を変更可能

**評価項目のスコープ:**
- 等級×職種小分類の単位での一括編集（例：正4×多能工の評価項目をまとめて設定）
- 個人単位での編集・修正
- 存在しない等級×職種の組み合わせには評価項目は設定不要

#### 2.6.2 360度評価

複数の評価者による多面的評価を実施する。

**設定項目:**
- 被評価者一覧（従業員データベースから取得）
- 各被評価者に対する評価者の割り当て（最大4名：評価者1〜評価者4）
- 評価期間の設定（上期：例 202511、下期：例 202605）
- 評価項目の設定（大項目・中項目の階層構造、行追加可能）

**評価フロー:**
1. 管理者が「誰が誰を評価するか」を設定
2. 評価依頼を一斉通知（メール or アプリ内通知）
3. 各評価者がスマホ or PCで評価を入力
4. 得点の自動集計（平均点の算出）
5. フィードバック（FB）欄への記入

**評価者割り当ての初期段階:**
- MVP段階では、表形式で被評価者の名前を並べ、各人に対して評価者を手動で割り当てる
- 将来的には配置システムと連携し、「誰と誰が一緒に仕事したか」から自動で推薦

### 2.7 ダッシュボード

#### 2.7.1 経営者向けダッシュボード

- 従業員の等級分布（等級×職種のマトリクス表示）
- 評価結果の全体サマリー（評価レート分布）
- 人件費の概算（基本給合計、昇給シミュレーション後の予測）
- 売上目標に対する人員状況の可視化
- 職種ごとの人数、等級ごとの人数
- 一人当たりの工事量（将来的に決算データ連携時）

#### 2.7.2 NiNKU BOXX運用者向けダッシュボード

- 全クライアント企業の一覧と進捗状況
- 各社の評価運用ステータス（未実施 / 評価中 / 集計完了 / FB完了）
- 号俸テーブルのミスマッチアラート一覧

## 3. データモデル（主要エンティティ）

```
Company（会社）
├── Department（部署）
├── JobCategory（職種大分類：例「現場」「管理」）
│   └── JobType（職種小分類：例「多能工」「ドライバー」「オペレーター」「解体D」）
├── Grade（等級：正1〜正6等）
│   └── GradeJobTypeConfig（等級×職種の有効/無効設定）
│       └── GradeRole（等級×職種別の役割責任・該当者）
├── SalaryTable（号俸テーブル）
│   └── SalaryTableEntry（号俸 × 等級の基本給）
```

## 5. 開発優先度（フェーズ分け）

### Phase 1：MVP（最小限のプロダクト）

等級制度が既に定義済みの企業が、評価運用を行えるようにする。

1. **会社管理** - 会社情報の登録、部署・職種・等級の基本設定
2. **従業員データベース** - 従業員の登録・一覧・編集（基本属性 + 等級・職種・基本給）
3. **号俸テーブル** - パラメータ入力によるテーブル自動生成、従業員の自動当てはめ
4. **個別評価** - 評価項目の設定、評価入力、スコア集計
5. **360度評価** - 評価者割当（手動テーブル方式）、評価入力、得点集計
6. **PDF出力** - 評価シートの印刷用PDF

### Phase 2：運用強化

1. **評価基準・昇降格ロジック** - 評価レートマトリクス、号俸改定基準の自動計算
2. **ダッシュボード** - 等級分布、評価結果サマリー、ミスマッチアラート
3. **通知機能** - 評価依頼の一斉メール/通知
4. **パーソナルシート** - 個人の情報集約ページ
5. **面談記録管理** - 面談日時・議事録リンクの時系列管理

### Phase 3：高度化・拡張

1. **人員配置シミュレーション** - 部署異動・配置のシミュレーション
2. **決算データ連携** - 売上・利益目標との人員状況の照合
3. **採用支援** - 等級テーブルを活用した採用時の給与提示
4. **給与連動** - 手当・賞与を含めた総報酬の管理
5. **モバイル最適化** - スマホからの評価入力UI最適化

## 6. 非機能要件

### 6.1 セキュリティ
- 従業員の個人情報・給与情報を扱うため、高いセキュリティが必要
- SSL/TLS暗号化
- ロールベースのアクセス制御（RBAC）
- 監査ログの記録

### 6.2 パフォーマンス
- 想定クライアント数：初期9社、将来的に数十社
- 1社あたりの従業員数：数十名〜数百名
- 号俸テーブルの生成・表示は即座にレスポンス

### 6.3 可用性
- SaaS型Webアプリケーション
- PCブラウザ + スマートフォンブラウザ対応
- 評価入力時期（年2回）にアクセスが集中する想定

### 6.4 データ移行
- 既存のスプレッドシートからのデータインポート機能（CSV/Excel）
- 初期データのバルクインポート対応

## 7. 技術的な検討事項

### 7.1 号俸テーブルのロジック変更リスク
- 議事録で言及されている通り、基準ロジックの変更が後から出た場合の影響範囲が大きい
- 号俸テーブルの生成ロジックは十分にパラメータ化し、柔軟に対応できる設計にする
- テーブル生成パラメータのバージョン管理を検討

### 7.2 制度設定の修正に伴う整合性管理
- 等級数やランク分割を変更した場合、既存の従業員データ（等級、号俸、ランク）との整合性を保つ仕組みが必要
- 号俸テーブル再生成時に、全従業員の号俸位置を自動再マッピングするか、手動で再割当するかの選択肢を提供
- 制度変更の「適用日」を設定可能にし、変更前と変更後の制度をそれぞれ保持する設計とする（スナップショット方式）
- 過去の評価結果は変更前の制度に基づいた値として保全し、遡及変更は原則行わない

### 7.3 給与データとの紐付け
- MVP段階では基本給のみ管理し、手当・賞与等の給与データとの完全連携はPhase 3以降
- ただし、将来的な連携を見据えたデータモデル設計が必要

### 7.4 等級制度の前提
- このプロダクトは「等級制度が定義済み」の状態から使うことを前提とする（Phase 1）
- ただし、制度の修正・調整は運用中いつでも可能であり、等級数の変更やランク構成の見直しにも対応する
- 等級制度がまだない企業への対応（制度設計ウィザード等）はPhase 2以降で検討
