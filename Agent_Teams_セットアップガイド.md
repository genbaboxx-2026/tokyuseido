# NiNKU BOXX プロジェクト — Claude Code Agent Teams セットアップガイド

## 1. Agent Teamsとは

Claude Codeの実験的機能で、複数のClaudeインスタンスがチームとして並行作業できる仕組みです。1つの「リード（チームリーダー）」が全体を統括し、複数の「チームメイト」がそれぞれ独立したコンテキストウィンドウで並行して作業します。チームメイト同士はメッセージで直接やり取りが可能です。

**通常のサブエージェントとの違い：**
- サブエージェント：親に結果を返すだけ、横の連携なし
- Agent Teams：チームメイト同士が直接メッセージをやり取りし、自律的に連携

## 2. セットアップ手順

### Step 1: Agent Teamsを有効化

Agent Teamsは実験的機能のため、デフォルトでは無効です。以下のいずれかで有効化します。

**方法A: settings.json に追加（推奨）**

`~/.claude/settings.json` を開いて以下を追加：

```json
{
  "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
}
```

**方法B: 環境変数で設定**

```bash
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
```

### Step 2: プロジェクトのCLAUDE.mdを作成

`tokyuseido/CLAUDE.md` をプロジェクトルートに配置します。Agent Teamsのリードエージェントとチームメイトが参照するプロジェクト指示書です。

```markdown
# NiNKU BOXX - 人事制度プロダクト

## プロジェクト概要
ブルーカラー企業向けの等級制度・評価制度構築Webアプリケーション。

## 技術スタック
- フロントエンド: Next.js (App Router) + TypeScript + Tailwind CSS
- バックエンド: Next.js API Routes (or別途検討)
- DB: PostgreSQL + Prisma ORM
- 認証: NextAuth.js
- 状態管理: Zustand or TanStack Query

## ディレクトリ構造
tokyuseido/
├── CLAUDE.md            # このファイル
├── NiNKU_BOXX_要件定義書.md  # 要件定義書
├── src/
│   ├── app/             # Next.js App Router pages
│   ├── components/      # UIコンポーネント
│   ├── lib/             # ユーティリティ・ビジネスロジック
│   ├── prisma/          # Prisma schema & migrations
│   └── types/           # TypeScript型定義
├── public/
└── tests/

## Agent Teams向けルール
- 各チームメイトは自分の担当領域のファイルのみ編集すること
- 共有の型定義（src/types/）を変更する場合は、リードに報告してから行う
- Prisma schemaの変更はDB担当のチームメイトのみが行う
- コンポーネントは必ずTypeScriptで記述する
- テストは各機能の実装と同時に作成する

## 主要エンティティ（DB設計の参考）
Company, Department, JobCategory, JobType, Grade,
GradeJobTypeConfig, GradeRole, SalaryTable, SalaryTableEntry,
Employee, EvaluationPeriod, IndividualEvaluation, Evaluation360,
EvaluatorAssignment, Position
```

### Step 3: Claude Codeを起動

```bash
cd APP/tokyuseido
claude
```

## 3. Agent Teamsの使い方（プロンプト例）

### 3.1 プロジェクト初期セットアップ

```
NiNKU_BOXX_要件定義書.md を読んでください。

このプロジェクトをAgent Teamsで構築します。以下のチーム構成で作業を開始してください：

チームメイト1（DB設計担当）:
- Prisma schemaの設計・実装
- マイグレーションの作成
- シードデータの作成
- 要件定義書のデータモデルセクションを参照

チームメイト2（API設計担当）:
- Next.js API Routesの設計・実装
- 各エンティティのCRUD API
- バリデーションロジック

チームメイト3（フロントエンド担当）:
- ページコンポーネントの実装
- UIコンポーネントの作成
- Tailwind CSSでのスタイリング

まずはPhase 1（MVP）の範囲で、会社管理と従業員データベースから始めてください。
```

### 3.2 特定機能の並行開発

```
号俸テーブル機能をAgent Teamsで実装してください：

チームメイト1（ロジック担当）:
- 号俸テーブル自動生成のアルゴリズム実装（src/lib/salary-table/）
- パラメータ（基本給MAX/MIN、ランク分割、増加率、初期号差）から
  テーブルを生成する計算ロジック
- 従業員の自動当てはめロジック
- ミスマッチ検出ロジック

チームメイト2（UI担当）:
- 号俸テーブル設定画面（パラメータ入力フォーム）
- 号俸テーブル確認画面（マトリクス表示）
- 差分プレビュー画面
- 従業員当てはめ結果の表示

チームメイト3（テスト担当）:
- ロジックのユニットテスト
- APIのインテグレーションテスト
- エッジケース（パラメータ変更時の再計算等）のテスト

ロジック担当が型定義を先に作り、UI担当とテスト担当に共有してください。
```

### 3.3 等級×職種マトリクスの実装

```
等級×職種の存在制御機能をAgent Teamsで実装してください：

チームメイト1（バックエンド）:
- GradeJobTypeConfig テーブルの実装
- 有効/無効の切り替えAPI
- 無効な組み合わせへの従業員割当をブロックするバリデーション
- 従業員が存在する組み合わせを無効にしようとした場合の警告API

チームメイト2（フロントエンド）:
- 等級×職種のマトリクス設定画面
- チェックボックスで有効/無効を切り替えるUI
- 警告モーダルの実装
- 役割責任編集画面（マトリクス表示、該当者の自動反映）

それぞれの担当領域のファイルのみ編集すること。
型定義はバックエンド担当が先に作成し、フロントエンド担当に共有。
```

### 3.4 評価制度の並行開発

```
評価制度をAgent Teamsで実装してください：

チームメイト1（個別評価）:
- 評価項目の設定API・UI
- 等級×職種の役割責任からのプリセット機能
- 評価入力フォーム（5段階スコア + フリーコメント）
- 評価スコアの集計

チームメイト2（360度評価）:
- 評価者割当の設定API・UI
- 評価入力画面（スマホ対応）
- 得点の自動集計（平均点）
- 評価依頼の一斉通知機能

チームメイト3（評価基準・昇降格）:
- 評価レートマトリクス（S/A/B/C/D）の設定
- 等級別号俸改定基準テーブルの設定
- 昇格・降格条件のチェックロジック
- 評価結果から号俸変動を算出するロジック

共通の型定義はチームメイト3が先に作成して共有。
```

## 4. 注意事項とベストプラクティス

### トークン消費について
- 各チームメイトが独立したコンテキストウィンドウを持つため、チームメイト数 × 通常のトークン量を消費する
- 3〜4チームメイトが現実的なバランス
- 単純な順次作業には単一セッションの方がコスパが良い

### Agent Teamsが効果的なケース
- フロントエンド・バックエンド・テストの並行開発
- 複数の独立した機能モジュールの同時実装
- コードレビュー（セキュリティ・パフォーマンス・テストカバレッジの各視点）
- 複数の仮説を並行して検証するデバッグ

### Agent Teamsを避けるべきケース
- 同じファイルの編集が必要な作業（コンフリクトの原因に）
- 依存関係が多い順次タスク
- 小さな修正や単純な変更

### ファイル競合を防ぐコツ
- CLAUDE.mdで「各チームメイトの担当ディレクトリ」を明確に指定する
- 共有の型定義は1人のチームメイトが先に作り、他のメンバーに共有
- Prisma schemaの変更は1人のチームメイトに限定

## 5. 推奨する開発の進め方

```
Phase 1 MVP の開発順序：

1回目のAgent Teams セッション:
  → DB設計 + プロジェクト初期セットアップ + 共通コンポーネント

2回目のAgent Teams セッション:
  → 会社管理機能（API + UI + テスト）

3回目のAgent Teams セッション:
  → 従業員データベース（API + UI + テスト）

4回目のAgent Teams セッション:
  → 号俸テーブル（ロジック + UI + テスト）

5回目のAgent Teams セッション:
  → 個別評価 + 360度評価（並行開発）

6回目のAgent Teams セッション:
  → PDF出力 + 全体結合テスト + バグ修正
```

各セッションの開始時に、要件定義書の該当セクションを指定してチームに読ませることで、正確な実装が可能になります。
