# 評価テンプレート → 従業員反映 改善 要件定義書

## 概要

評価実行画面（個別評価・360度評価の両方）において、テンプレートを編集・保存した後に従業員の評価項目へ反映する仕組みを改善する。

### 現状の課題

1. **個別評価**: テンプレート保存後にダイアログ内から「従業員に反映」できるボタンは追加済みだが、保存時の `status`（draft/confirmed）がAPIに送信されていないバグがある
2. **360度評価**: テンプレートを編集・保存しても、既に項目が生成済みの従業員には反映する手段がない。一括反映APIも存在しない
3. **360度評価**: テンプレート編集ダイアログのUIが古い（カテゴリ名+テキストエリア形式）

---

## Phase 1: 個別評価テンプレートのステータス保存バグ修正

### 1.1 問題の詳細

`IndividualTemplatePanel.tsx` の `handleSaveGradeTemplate` 関数で、引数として `status: "draft" | "confirmed"` を受け取っているが、APIリクエストの body に `status` を含めていない。

### 1.2 対象ファイル

- `src/components/operations/individual/IndividualTemplatePanel.tsx`

### 1.3 修正内容

`handleSaveGradeTemplate` 内の PUT リクエスト body に `status` を追加する。

**修正箇所（期間固有テンプレートが既にある場合の PUT）:**

```typescript
// 現在
body: JSON.stringify({
  name: templateName,
  items: validItems.map(...)
})

// 修正後
body: JSON.stringify({
  name: templateName,
  status, // "draft" or "confirmed" を追加
  items: validItems.map(...)
})
```

**修正箇所（マスターからコピー後の PUT）:**

同様に `status` を body に追加。

### 1.4 テスト観点

- 「保存」ボタン → status が "draft" で保存されること
- 「確定」ボタン → status が "confirmed" で保存されること
- 保存後にダイアログを再度開いた時にステータスが反映されていること

## Phase 2: 360度評価 - 一括反映APIの作成

### 2.1 概要

360度テンプレートを編集・保存した後に、対象の全従業員の評価項目（`EvaluationCustomItem`）を一括で更新するAPIを作成する。

### 2.2 新規APIエンドポイント

```
POST /api/companies/[id]/operations/[periodId]/360-templates/[templateId]/apply
```

### 2.3 リクエスト

```typescript
{
  overwrite?: boolean  // true: スコア入力済みの従業員も上書き、false: スキップ（デフォルト: false）
  employeeIds?: string[] // 指定時はその従業員のみ対象、未指定時は全対象従業員
}
```

### 2.4 処理フロー

```
1. 期間固有テンプレートを取得（カテゴリ・アイテム込み）
   - テーブル: Evaluation360Template + categories + items
   - 条件: templateId, periodId が一致

2. テンプレートの対象等級・職種を取得
   - テーブル: Evaluation360TemplateGrade, Evaluation360TemplateJobType

3. 対象従業員の360度レコードを取得
   - テーブル: Evaluation360Record
   - 条件: evaluationPeriodId = periodId
           AND employee.gradeId IN (テンプレート対象等級)
           AND employee.jobTypeId IN (テンプレート対象職種)
           AND status IN ("draft", "preparing_items", "preparing_reviewers", "ready")
   - employeeIds指定時はさらにフィルタ

4. 各従業員に対してトランザクション処理:
   a. 既存のEvaluationCustomItemを取得
      - 条件: employeeId, periodId, evaluationType="360"
   
   b. スコア入力チェック
      - Evaluation360Score にスコアが存在するか確認
      - overwrite=false かつスコアあり → スキップ
   
   c. 既存のEvaluationCustomItemを削除
      - DELETE WHERE employeeId, periodId, evaluationType="360"
   
   d. テンプレートからEvaluationCustomItemを新規作成
      - sourceTemplateItemId: テンプレート項目のID
      - itemName: テンプレート項目のcontent
      - maxScore: テンプレート項目のmaxScore
      - categoryName: カテゴリ名
      - categorySortOrder: カテゴリのsortOrder
      - sortOrder: 項目のsortOrder
      - isCustomized: false, isAdded: false, isDeleted: false

5. 結果を返す
```

### 2.5 レスポンス

```typescript
{
  success: true,
  appliedCount: number,    // 反映した従業員数
  skippedCount: number,    // スキップした従業員数
  message: string,         // "X名の従業員に反映しました"
  details: Array<{
    employeeId: string,
    status: "applied" | "skipped" | "error",
    reason?: string
  }>
}
```

### 2.6 対象ファイル

- **新規作成**: `src/app/api/companies/[id]/operations/[periodId]/360-templates/[templateId]/apply/route.ts`

### 2.7 DBスキーマ

変更なし。既存のテーブルのみ使用。

| テーブル | 操作 |
|----------|------|
| `Evaluation360Template` | 読み取り（テンプレート取得） |
| `Evaluation360TemplateCategory` | 読み取り（カテゴリ取得） |
| `Evaluation360TemplateItem` | 読み取り（項目取得） |
| `Evaluation360TemplateGrade` | 読み取り（対象等級取得） |
| `Evaluation360TemplateJobType` | 読み取り（対象職種取得） |
| `Evaluation360Record` | 読み取り（対象従業員取得） |
| `EvaluationCustomItem` | 削除 → 新規作成（従業員項目再生成） |
| `Evaluation360Score` | 読み取り（スコア入力チェック） |

### 3.4 UI仕様の詳細

1. **ダイアログサイズ**: フルスクリーン（`!max-w-[calc(100vw-80px)] w-[calc(100vw-80px)] h-[90vh]`）
2. **ヘッダー**: テンプレート名、説明文（この期間のみの変更であることを明示）
3. **カテゴリセクション**: 折りたたみ可能（ChevronDown/ChevronRight）
4. **カテゴリヘッダー**: カテゴリ名（編集可能）、項目数/合計満点のBadge、「従業員に反映」「保存」「確定」ボタン
   - 「従業員に反映」ボタンは Phase 2 の API を呼ぶ（テンプレート全体を対象）
   - 360度は個別評価と違い、1テンプレートが複数等級×複数職種にまたがるため、「テンプレート単位」で反映
5. **項目リスト**: No | 項目名（Input, h-8） | 満点（Input number, w-14） | 削除ボタン（Trash2 icon）
6. **操作ボタン**: 「+ 項目追加」（各カテゴリ内）、「+ カテゴリを追加」（セクション外）
7. **フッター**: 「閉じる」ボタン

### 3.5 保存の動作仕様

- 「保存」: カテゴリ単位ではなく**テンプレート全体を一括保存**（360度テンプレートはカテゴリ横断で1つ）
  - PUT `/api/companies/{companyId}/operations/{periodId}/360-templates/{templateId}`
  - body: `{ categories: [...], status: "draft" }`
- 「確定」: 同様に一括保存 + `status: "confirmed"`
- 「従業員に反映」: Phase 2 の apply API を呼ぶ
  - POST `/api/companies/{companyId}/operations/{periodId}/360-templates/{templateId}/apply`
  - body: `{ overwrite: false }`

**注意**: 360度テンプレートの保存は「テンプレート全体」が単位。個別評価のような「等級ごと」の保存ではない。理由は、360度テンプレートは1テンプレートが複数等級・職種にまたがるため。

### 3.6 「従業員に反映」ボタンの配置

360度はテンプレート全体が1単位なので、ボタンは**ダイアログフッター（閉じるボタンの左）に1つだけ**配置する。

```
│──────────────────────────────────────────────── │
│ [従業員に反映]                         [閉じる]  │
└─────────────────────────────────────────────────┘
```

## Phase 4: 360度評価の既存 bulk-generate-items API 改善

### 4.1 概要

現在の `bulk-generate-items` は `draft` ステータスかつ項目未設定の従業員のみが対象。これを改善して、`preparing_items` ステータスの従業員にも再生成できるようにする。

### 4.2 対象ファイル

- **修正**: `src/app/api/companies/[id]/operations/[periodId]/360/bulk-generate-items/route.ts`

### 4.3 修正内容

1. 対象ステータスを `draft` から `draft`, `preparing_items` に拡張
2. リクエスト body に `overwrite` フラグを追加（デフォルト false）
3. `overwrite: true` の場合は既に項目がある従業員にも再生成を実行

```typescript
// 現在
const draftRecords = await prisma.evaluation360Record.findMany({
  where: {
    companyId,
    evaluationPeriodId: periodId,
    status: "draft",
  },
  ...
})

// 修正後
const body = await request.json().catch(() => ({}))
const { overwrite = false } = body as { overwrite?: boolean }

const targetRecords = await prisma.evaluation360Record.findMany({
  where: {
    companyId,
    evaluationPeriodId: periodId,
    status: { in: ["draft", "preparing_items"] },
  },
  ...
})

// overwrite=false の場合は項目未設定のみ
// overwrite=true の場合は全対象
```

### 4.4 フロントエンド連携

`Eval360PreparingTab.tsx` の「一括テンプレ生成」ボタン（現在 disabled）を有効化し、`bulk-generate-items` API を呼び出すようにする。

## DBスキーマ（参考）

### 変更なし - 既存テーブルのみ使用

**個別評価系:**
- `PeriodEvaluationTemplate` - 期間固有テンプレート（@@unique [periodId, gradeId, jobTypeId]）
- `PeriodEvaluationTemplateItem` - 期間固有テンプレート項目
- `EmployeeEvaluation` - 従業員評価
- `EmployeeEvaluationItem` - 従業員評価項目（@@unique [employeeEvaluationId, evaluationTemplateItemId]）

**360度評価系:**
- `Evaluation360Template` - 360度テンプレート（@@unique [periodId, sourceTemplateId]）
- `Evaluation360TemplateCategory` - テンプレートカテゴリ
- `Evaluation360TemplateItem` - テンプレート項目
- `Evaluation360TemplateGrade` - テンプレート等級紐付け（@@unique [templateId, gradeId]）
- `Evaluation360TemplateJobType` - テンプレート職種紐付け（@@unique [templateId, jobTypeId]）
- `Evaluation360Record` - 360度評価レコード（@@unique [evaluationPeriodId, employeeId]）
- `EvaluationCustomItem` - カスタム評価項目（@@unique [employeeId, periodId, evaluationType, sourceTemplateItemId]）
- `Evaluation360Score` - 360度評価スコア

## 注意事項

### 個別評価 vs 360度評価の構造的な違い

| | 個別評価 | 360度評価 |
|---|---|---|
| テンプレート単位 | 等級×職種ごとに1テンプレート | 1テンプレートが複数等級×職種にまたがる |
| 従業員項目テーブル | `EmployeeEvaluationItem` | `EvaluationCustomItem` |
| テンプレート項目構造 | フラット（name, category, maxScore） | カテゴリ→項目の2層構造 |
| スコア確認先 | `EmployeeEvaluationItem.selfScore / evaluatorScore` | `Evaluation360Score` テーブル |
| 反映の単位 | 等級ごと（ダイアログ内の各セクション） | テンプレート全体（ダイアログのフッター） |

### overwrite フラグの扱い

- `overwrite: false`（デフォルト）: 既にスコアが入力されている従業員はスキップ
- `overwrite: true`: スコア入力済みでも上書き（データ消失の可能性あり、確認ダイアログ推奨）
- フロントエンドでは常に `overwrite: false` で呼び出す
- 将来的に確認ダイアログ付きで `overwrite: true` を選択可能にすることを検討

### EvaluationCustomItem の unique 制約

`@@unique([employeeId, periodId, evaluationType, sourceTemplateItemId])` があるため、同じテンプレート項目を同じ従業員に2回作成するとエラーになる。反映時は必ず既存レコードを削除してから新規作成すること。
