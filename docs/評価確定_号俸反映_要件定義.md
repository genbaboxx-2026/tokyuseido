# 評価確定・号俸反映 要件定義書

## 1. 概要

### 1.1 背景と課題

現在の評価実行ページ（`/companies/[companyId]/operations/[periodId]`）では、「個別評価」と「360度評価」がそれぞれ独立したワークフロー（準備→配布・回収→集計→完了）を持っている。

しかし、両方の評価が完了した後の **「結果統合→ランク決定→号俸反映」** の導線が存在しない。要件定義書(2.5.1)で定義されている評価の全体フロー（Step 2〜5）が未実装のまま、評価結果が宙に浮いた状態になっている。

```
Step 1: 評価実施（360度評価 + 個別評価）  ← ここまでしか実装されていない
    ↓
Step 2: 割合設定で100点満点に換算          ← 未実装
    ↓
Step 3: ランク閾値でS/A/B/C/D決定         ← 未実装
    ↓
Step 4: 前期×後期マトリクス（年2回のみ）   ← 未実装
    ↓
Step 5: 等級別号俸改定基準で号俸変動       ← 未実装
```

### 1.2 解決方針

評価実行ページを **2ステップ構成** に再構成する。

- **Step 1: 評価実行** — 現在のページをアコーディオン形式にリファクタ。両方の評価が完了するとアコーディオンが閉じ、「最終確認を行う」ボタンが出現
- **Step 2: 最終確認・号俸反映** — 新規ページ。結果統合→ランク決定→号俸変動算出→最終確認・確定を一貫して行う

確定時に従業員データ（号俸・ランク・基本給）と号俸テーブル上の配置を自動更新し、評価期間を「完了」に変更する。

---

## 2. 全体構成

```
評価期間ページ（2ステップ制）
│
├── Step 1: 評価実行
│   ├── 📋 個別評価（アコーディオンカード）
│   │   └── 準備 → 配布・回収 → 集計 → 完了
│   ├── 👥 360度評価（アコーディオンカード）
│   │   └── 準備 → 配布・回収 → 集計 → 完了
│   │
│   └── ✅ 両方完了 → アコーディオン閉じる
│       └── [ 🏆 最終確認を行う ] ボタン出現
│
└── Step 2: 最終確認・号俸反映（新規ページ）
    ├── ① 結果統合（360度 + 個別 → 100点満点）
    ├── ② ランク決定（S/A/B/C/D）
    ├── ③ 号俸変動算出（改定基準テーブル適用）
    └── ④ 最終確認 → 確定ボタン
        └── 従業員データ反映 + 号俸テーブル反映
```

---

## 3. 用語定義

| 用語 | 意味 |
|------|------|
| 結果統合 | 360度評価と個別評価の得点を割合で按分し、100点満点の合計点を算出する処理 |
| 評価ランク | 合計点をランク閾値で変換したS/A/B/C/Dの評価 |
| 年間ランク | 年2回実施の場合、前期×後期マトリクスで決定される最終ランク。年1回の場合は評価ランクと同一 |
| 号俸変動 | 年間ランク×現在の号俸帯から算出される号俸の増減値（+N / -N / 0） |
| 確定 | 最終確認後に号俸変動を従業員データに反映する操作。確定後は原則として取り消し不可 |

---

## 4. 機能要件

### 4.1 Step 1: 評価実行ページのリファクタ

#### 4.1.1 アコーディオンカード化

現在の個別評価セクションと360度評価セクションを、開閉可能なアコーディオンカードに変更する。

**アコーディオンカードの構成:**

```
┌─ 📋 個別評価  完了: 2/5  ──────────────────── [▼] ─┐
│                                                      │
│  [準備] → [配布・回収] → [集計] → [完了]              │
│                                                      │
│  （従業員リスト・操作UI: 既存のまま）                   │
│                                                      │
└──────────────────────────────────────────────────────┘
```

**カードヘッダーに表示する情報:**
- 評価種別アイコン + 名称（📋 個別評価 / 👥 360度評価）
- 完了状況バッジ（完了: N/M）
- 完了済みの場合は ✅ アイコン
- 「対象者追加」ボタン（ヘッダー右側、既存機能）
- アコーディオン開閉トグル

**開閉の挙動:**

| 状態 | 個別評価 | 360度評価 | 挙動 |
|------|---------|----------|------|
| どちらも進行中 | 開 | 開 | 両方開いた状態 |
| 片方のみ完了 | 完了→閉 | 進行中→開 | 完了した方は自動で閉じる（手動で開閉可能） |
| 両方完了 | 閉 | 閉 | 両方自動で閉じる（手動で開閉可能） |

#### 4.1.2 「最終確認を行う」ボタン

両方の評価が完了した場合にのみ表示される導線。

```
┌──────────────────────────────────────────────────────┐
│  🎉 すべての評価が完了しました                          │
│                                                        │
│  評価結果を統合し、号俸変動の最終確認に進んでください。    │
│                                                        │
│                 [ 🏆 最終確認を行う → ]                  │
└──────────────────────────────────────────────────────┘
```

**表示条件:**
- 個別評価の全員が `status === "COMPLETED"`（EvaluationStatus enum）
- 360度評価の全員が `status === "completed"`（Stringフィールド）
- ※ 片方しか評価種別がない場合は、その種別のみ完了していればOK

**ボタン押下時:**
- Step 2 ページ（`/companies/[companyId]/operations/[periodId]/finalize`）に遷移
- 遷移時、バックエンドで `EvaluationResult` レコードの自動生成（未存在の場合）

#### 4.1.3 ステップインジケーター

ページ上部に2ステップのインジケーターを表示する。

```
  [ ① 評価実行 ] ──→ [ ② 最終確認・号俸反映 ]
      ●                      ○
```

- Step 1 がアクティブの場合: Step 1 にアクセント色
- Step 2 がアクティブの場合: Step 1 に完了チェック、Step 2 にアクセント色
- Step 2 はクリックで遷移可能（ただし両方の評価が完了している場合のみ）

### 4.2 Step 2: 最終確認・号俸反映ページ

新規ページ: `/companies/[companyId]/operations/[periodId]/finalize`

1ページ内で4つのサブステップを上から順に表示するウィザード風UI。各サブステップは折りたたみ可能で、上から順に進めていく。

#### 4.2.1 サブステップ① 結果統合

360度評価と個別評価の得点を割合で按分し、100点満点の合計点を算出する。

**表示内容:**

```
━━━ ① 結果統合 ━━━━━━━━━━━━━━━━━━━━━━━━━━

  割合設定: 360度 70% / 個別 30%  [割合設定を確認 ↗]

  ┌────────────────────────────────────────────────────────┐
  │ 従業員名   │ 等級 │ 職種     │ 360度得点  │ 個別得点  │ 合計点  │
  │────────────│──────│──────────│───────────│──────────│─────────│
  │ 佐藤 花子  │ 正2  │ 経営管理 │ 325/350   │ 140/150  │  93.0   │
  │ 伊藤 雄一  │ 正3  │ 多能工   │ 310/350   │ 130/150  │  88.0   │
  │ 中村 隆    │ 正4  │ オペ     │ 280/350   │ 120/150  │  80.0   │
  │ 田中 太郎  │ 正4  │ ドライバー│ 240/350   │ 90/150   │  66.0   │
  │ 山田 次郎  │ 正5  │ 多能工   │ 200/350   │ 60/150   │  52.0   │
  └────────────────────────────────────────────────────────┘
```

**計算式:**

```
合計点 = (360度得点 / 360度満点 × 360度割合) + (個別得点 / 個別満点 × 個別割合)
```

**割合の取得元:**
1. `EmployeeEvaluationWeight` に個人別設定がある場合 → そちらを優先
2. なければ会社のデフォルト割合を使用

**360度のみ / 個別のみの従業員:**
- 片方の評価しか持たない従業員は、持っている評価のみで100点満点に換算
- （例: 360度のみ → 360度得点 / 360度満点 × 100）

**表示の補足:**
- 個人別に割合が異なる場合、該当行に🔧アイコンで表示
- 合計点でソート可能（デフォルトは高い順）
- ヘッダーに「割合設定を確認」リンク（評価制度設定ページの割合設定タブへ）

合計点にランク閾値を適用し、S/A/B/C/Dを決定する。

**表示内容:**

```
━━━ ② ランク決定 ━━━━━━━━━━━━━━━━━━━━━━━━━━

  ランク閾値: S≧95 / A≧85 / B≧70 / C≧50 / D<50  [閾値設定を確認 ↗]

  ┌─────────────────────────────────────────────────────┐
  │ 従業員名   │ 合計点 │ ランク │                           │
  │────────────│────────│────────│───────────────────────────│
  │ 佐藤 花子  │  93.0  │   A    │ ██████████████████████░░░ │
  │ 伊藤 雄一  │  88.0  │   A    │ ████████████████████░░░░░ │
  │ 中村 隆    │  80.0  │   B    │ ████████████████░░░░░░░░░ │
  │ 田中 太郎  │  66.0  │   C    │ █████████████░░░░░░░░░░░░ │
  │ 山田 次郎  │  52.0  │   C    │ ██████████░░░░░░░░░░░░░░░ │
  └─────────────────────────────────────────────────────┘

  分布サマリ:  S: 0名  A: 2名  B: 1名  C: 2名  D: 0名
```

**閾値の取得元:**
- `EvaluationScoringMethod` + `EvaluationRank` テーブルから取得
- 存在しない場合はデフォルト値（S≧95, A≧85, B≧70, C≧50, D<50）

**年2回実施の場合（前期×後期マトリクス）:**

該当の評価期間が `SECOND_HALF`（後期）であり、同一会社に同年度の `FIRST_HALF`（前期）の確定済み結果が存在する場合:

```
  前期ランク × 後期ランク → 年間ランク（EvaluationCriteriaマトリクス適用）

  ┌───────────────────────────────────────────────────────────┐
  │ 従業員名   │ 前期ランク │ 後期ランク │ 年間ランク │           │
  │────────────│────────────│────────────│────────────│           │
  │ 佐藤 花子  │     A      │     A      │     A      │           │
  │ 伊藤 雄一  │     B      │     A      │     B      │           │
  │ ...        │            │            │            │           │
  └───────────────────────────────────────────────────────────┘
```

- `EvaluationCriteria` テーブル（`firstHalfRating × secondHalfRating → finalRating`）を使用
- 前期結果が存在しない場合はこのステップをスキップし、今期ランクをそのまま使用

#### 4.2.3 サブステップ③ 号俸変動算出

年間ランク（または今期ランク）×現在の号俸帯から、等級別号俸改定基準を適用して号俸変動を算出する。

**表示内容:**

```
━━━ ③ 号俸変動 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  等級別号俸改定基準を適用  [改定基準を確認 ↗]

  ┌──────────────────────────────────────────────────────────────┐
  │ 従業員名  │ 等級│ランク│現号俸帯│現号俸│変動 │新号俸│ 現基本給  →  新基本給 │
  │───────────│─────│──────│────────│──────│─────│──────│───────────────────── │
  │ 佐藤 花子 │ 正2 │  A   │  B3    │  45  │ +2  │  47  │ 285,000 → 291,000  │
  │ 伊藤 雄一 │ 正3 │  A   │  C1    │  33  │ +1  │  34  │ 245,000 → 248,000  │
  │ 中村 隆   │ 正4 │  B   │  B5    │  43  │  0  │  43  │ 238,000 → 238,000  │
  │ 田中 太郎 │ 正4 │  C   │  A2    │  54  │ -2  │  52  │ 262,000 → 256,000  │
  │ 山田 次郎 │ 正5 │  C   │  D4    │  12  │ -1  │  11  │ 198,000 → 196,100  │
  └──────────────────────────────────────────────────────────────┘

  人件費影響: 月額合計 +3,100円 / 年額合計 +37,200円

  変動サマリ: ↑昇号: 2名 / →据え置き: 1名 / ↓降号: 2名
```

**算出ロジック:**

```
1. 従業員の等級(gradeId)と現在の号俸帯(currentRank)を取得
2. GradeAdjustmentRule から (gradeId, currentRank, rating) → stepAdjustment を取得
3. 新号俸 = 現号俸(currentStep) + stepAdjustment
4. 新号俸に対応する新ランクと新基本給を SalaryTableEntry から取得
5. 号俸の上限/下限チェック（号俸1未満にはならない、最大号俸を超えない）
```

**改定基準が未設定の場合:**
- 該当する等級×ランクの改定基準が存在しない従業員は「⚠ 改定基準未設定」と表示
- 号俸変動は0（据え置き）として扱う
- 「改定基準を設定」リンクで号俸テーブルの改定基準タブに遷移可能

**従業員に号俸が設定されていない場合:**
- `currentStep` がnullの従業員は「⚠ 号俸未設定」と表示
- 号俸変動の計算対象外（スキップ）
- 「従業員設定を確認」リンクで従業員編集ページに遷移可能

**人件費影響の計算:**
- 月額影響 = Σ(新基本給 - 現基本給)
- 年額影響 = 月額影響 × 12

### 4.3 確定時の反映処理

確定ボタン押下時、1つのトランザクション内で以下を全て実行する。

#### 4.3.1 反映先と更新内容

| # | 反映先 | テーブル | 更新内容 |
|---|--------|---------|---------|
| 1 | 従業員の号俸・ランク・基本給 | `Employee` | `currentStep`, `currentRank`, `baseSalary` を新しい値に更新 |
| 2 | 給与履歴 | `EmployeeSalary` | 新レコードを作成（effectiveDate=確定日, baseSalary=新基本給, reason="評価期間: {期間名}"） |
| 3 | 号俸テーブル上の配置 | `EmployeeCurrentSalary` | `currentSalary` を新基本給に更新 |
| 4 | 評価結果レコード | `EvaluationResult` | `status` を `CONFIRMED` に更新、`confirmedAt` を設定 |
| 5 | 評価期間 | `EvaluationPeriod` | `status` を `COMPLETED` に更新 |

#### 4.3.2 反映のルール

- **号俸の上下限**: 新号俸が1未満の場合は1に、最大号俸を超える場合は最大号俸にクランプ
- **号俸未設定者**: `currentStep` がnullの従業員はスキップ（Employee更新なし、EmployeeSalaryレコードも作成しない）
- **改定基準未設定**: 該当する `GradeAdjustmentRule` が存在しない場合は号俸変動0として扱う
- **等級変更は行わない**: この処理では号俸のみを変更し、等級の昇降格は別途管理者が判断する
- **確定後の取り消し**: 原則として取り消し不可。ただし、管理者が `EvaluationResult` を `REVERTED` に変更して従業員データを手動修正することは可能（将来的にUIを提供）

#### 4.3.3 エラーハンドリング

| エラー条件 | 対応 |
|-----------|------|
| トランザクション失敗 | 全更新をロールバック、エラーメッセージを表示 |
| 号俸テーブルが存在しない | 確定ボタンを無効化、警告メッセージを表示 |
| 改定基準が一部未設定 | 警告を表示した上で確定可能（未設定者は据え置き） |
| 既に確定済みの期間 | 確定ボタンを無効化、確定済みメッセージを表示 |

---

### 4.4 片方の評価種別のみの場合

すべての従業員が両方の評価を持つとは限らない。以下のパターンに対応する。

| パターン | 結果統合の計算 |
|---------|--------------|
| 360度 + 個別 両方あり | 割合で按分して100点満点 |
| 360度のみ | 360度得点 / 360度満点 × 100 |
| 個別のみ | 個別得点 / 個別満点 × 100 |
| どちらもなし | 対象外（テーブルに表示しない） |

**「最終確認を行う」ボタンの表示条件（詳細）:**
- 評価期間に個別評価対象者がいる場合 → 全員が完了していること
- 評価期間に360度評価対象者がいる場合 → 全員が完了していること
- どちらの対象者もいない場合 → ボタンは表示しない

## 5. データモデル

### 5.1 新規テーブル

#### EvaluationResult（評価確定結果）

従業員×評価期間ごとの最終評価結果と号俸変動を保存する。

| カラム | 型 | 説明 |
|--------|------|------|
| id | String (CUID) | PK |
| evaluationPeriodId | String | FK → EvaluationPeriod |
| employeeId | String | FK → Employee |
| score360Raw | Float? | 360度評価の素点 |
| score360Max | Float? | 360度評価の満点 |
| scoreIndividualRaw | Float? | 個別評価の素点 |
| scoreIndividualMax | Float? | 個別評価の満点 |
| ratio360 | Float? | 360度評価の割合（%） |

## 6. API設計

### 6.1 評価結果

| メソッド | エンドポイント | 説明 |
|---------|--------------|------|
| POST | `/api/companies/[id]/operations/[periodId]/finalize/generate` | 評価結果をDRAFT状態で生成（計算実行） |
| GET | `/api/companies/[id]/operations/[periodId]/finalize` | 評価結果一覧を取得（計算済みデータ） |
| POST | `/api/companies/[id]/operations/[periodId]/finalize/confirm` | 評価結果を確定し、従業員データに反映 |
| POST | `/api/companies/[id]/operations/[periodId]/finalize/recalculate` | 評価結果を再計算（割合変更後など） |

### 6.2 API詳細

#### POST `/finalize/generate`

**処理:**
1. 評価期間に紐づく全従業員の完了済み評価を収集
2. 各従業員の割合設定（`EmployeeEvaluationWeight`）を取得
3. 合計点を算出（100点満点）
4. ランク閾値を適用してランクを決定
5. 前期結果が存在する場合は年間ランクマトリクスを適用
6. `GradeAdjustmentRule` から号俸変動を算出
7. `SalaryTableEntry` から新号俸・新ランク・新基本給を取得
8. `EvaluationResult` レコードを `DRAFT` 状態で保存

**レスポンス:**

```typescript
interface GenerateFinalizeResponse {
  results: EvaluationResultData[]
  summary: {
    totalEmployees: number
    rankDistribution: Record<EvaluationRating, number>
    stepChanges: { up: number; same: number; down: number }
    monthlyCostImpact: number
    annualCostImpact: number
    missingAdjustmentRules: number
    missingStepEmployees: number
  }
}

interface EvaluationResultData {
  id: string
  employee: {
    id: string
    firstName: string
    lastName: string
    grade: { id: string; name: string } | null
    jobType: { id: string; name: string } | null
  }
  score360Raw: number | null
  score360Max: number | null
  scoreIndividualRaw: number | null
  scoreIndividualMax: number | null
  ratio360: number | null
  ratioIndividual: number | null
  combinedScore: number
  periodRank: EvaluationRating
  previousPeriodRank: EvaluationRating | null
  annualRank: EvaluationRating
  previousStep: number | null
  previousRank: string | null
  previousBaseSalary: number | null
  stepAdjustment: number
  newStep: number | null
  newRank: string | null
  newBaseSalary: number | null
  warnings: string[]  // "改定基準未設定", "号俸未設定" など
}
```

#### POST `/finalize/confirm`

**処理:**
1. `EvaluationResult` が全て `DRAFT` 状態であることを確認
2. トランザクション内で:
   - 各従業員の `Employee` を更新（currentStep, currentRank, baseSalary）
   - 各従業員の `EmployeeSalary` に新レコードを作成
   - 各従業員の `EmployeeCurrentSalary` を更新
   - 全 `EvaluationResult` の status を `CONFIRMED` に更新
   - `EvaluationPeriod` の status を `COMPLETED`、`finalizedAt` と `finalizedBy` を設定
3. 成功時にレスポンスを返却

**レスポンス:**

```typescript
interface ConfirmFinalizeResponse {
  success: boolean
  confirmedAt: string
  updatedEmployees: number
  skippedEmployees: number  // 号俸未設定でスキップした従業員数
}
```

## 8. 処理フロー

### 8.1 Step 1 → Step 2 遷移フロー

```
1. ユーザーが各評価を完了まで進める
2. 個別評価・360度評価の両方が完了
3. アコーディオンが自動で閉じる
4. 「🎉 すべての評価が完了しました」バナーが表示
5. 「最終確認を行う」ボタンをクリック
6. /finalize ページに遷移
7. バックエンドで POST /finalize/generate が呼ばれ、EvaluationResult を生成
8. 計算結果が4つのサブステップとして表示
```

### 8.2 結果統合〜確定フロー

```
1. ① 結果統合: 自動計算された合計点一覧を確認
2. ② ランク決定: ランク閾値適用結果を確認
3. ③ 号俸変動: 改定基準適用結果を確認
   - 改定基準未設定の警告がある場合は対応を検討
4. ④ 最終確認: サマリを確認
5. 「確定して反映する」ボタンをクリック
6. 確認ダイアログ表示（「本当に確定しますか？この操作は取り消せません。」）
7. 「確定する」で POST /finalize/confirm が実行
8. トランザクション内で全更新が実行
9. 成功画面が表示
10. 評価期間のステータスが「完了」に変更
```

### 8.3 再計算フロー

```
1. Step 2 表示中に、割合設定や改定基準を変更したい場合
2. 「割合設定を確認 ↗」リンクから設定ページに遷移して変更
3. Step 2 に戻る
4. 「再計算」ボタンをクリック
5. POST /finalize/recalculate で全結果を再計算
6. EvaluationResult を更新（DRAFT状態のまま）
7. 画面が最新の計算結果で更新
```

## 11. 注意事項・制約

- **トランザクション**: 確定処理は1つのトランザクション内で実行し、部分的な更新を防ぐ
- **べき等性**: `/finalize/generate` は何度呼んでも同じ結果を返す（既存のDRAFTレコードがあれば再計算して上書き）
- **同時実行防止**: 確定処理は楽観ロックで保護（`EvaluationResult.updatedAt` チェック）
- **パフォーマンス**: 従業員数が多い場合（100名超）はバッチ処理を検討
- **後方互換**: `EvaluationResult` がない評価期間は従来通り動作する
- **等級変更は対象外**: この処理では号俸変動のみ。昇格・降格の判断は管理者が別途行う
- **評価が一部のみの従業員**: 360度のみ、個別のみの従業員も正しく処理できるようにする
